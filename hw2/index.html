Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau’s algorithm evaluates a Bezier curve at a parameter t by repeatedly linearly interpolating between adjacent control points. Starting from the original control points p0…pn, one subdivision step produces a new list of points p'i = (1−t)·pi + t·p{i+1}. Repeating this process reduces the list size by 1 each level until only one point remain and that final point is the point on the Bezier curve at parameter t.

In my implementation, BezierCurve::evaluateStep(points) performs exactly one level of this interpolation where it loops over each adjacent pair (points[i], points[i+1]) and pushes (1−t)·points[i] + t·points[i+1] into an output vector (size n−1).

Part 2: Bezier surfaces with separable 1D de Casteljau

De Casteljau’s algorithm extends to Bezier surfaces by treating the Bezier surface as two nested 1D Bezier evaluations (the surface is separable in u and v). Given a 2D grid of control points P(i,j), we first run 1D de Casteljau along one parameter direction to collapse the grid into a single set of intermediate control points, then run 1D de Casteljau again along the other parameter to obtain the final point on the surface at (u,v). (Evaluating u first then v, or v first then u, produces the same surface point.)

In my implementation, BezierPatch::evaluateStep(points, t) performs one de Casteljau step in 3D by linearly interpolating adjacent points: p'i = (1−t)·pi + t·p{i+1}. Then BezierPatch::evaluate1D(points, t) repeatedly calls evaluateStep until only one point remains.

To evaluate the full surface, BezierPatch::evaluate(u, v) applies this 1D routine twice: it evaluates u on each of the 4 Bezier curves in that direction to produce 4 intermediate points, then evaluates v on those 4 points to return the final surface point.

Part 3: Area-weighted vertex normals

To implement area-weighted vertex normals, we used the half-edge traversal pattern from the half-edge primer, specifically the printNeighborPositions example which iterates around a vertex using h = h->twin()->next(). For each face, we retrieved the triangle's three vertex positions via h->vertex(), h->next()->vertex(), and h->next()->next()->vertex(). We then computed the cross product for each triangle and accumulated the results. The cross product, as noted in the spec, produces a vector whose direction is the face normal and whose magnitude equals twice the triangle area (\(\text{area} = \frac{1}{2} |u| \cdot |v| \cdot \sin\theta\), so \(|\text{cross}(u, v)| = 2 \cdot \text{area}\)). We then normalized the accumulated vector by dividing by its norm to get a unit normal.

Part 4: Edge flip

We largely followed the CMU Halfedge Edge Op Implementation Guide which outlines the approach: draw before/after diagrams, collect all elements, then exhaustively reassign every pointer. We filled in the parts left open in the guide which were the remaining halfedge reassignments (h2–h8), edges, and faces. The trickiest part was keeping track of which vertex maps to which variable, since there are 10 halfedges, 4 vertices, 5 edges, and 2 faces to manage. One common bug we ran into was accidentally swapping the vertex assignments, for example, confusing which vertex h0->vertex() returns vs h1->vertex(), since a halfedge's vertex is its source, not its destination. This led to faces with the wrong winding order, which caused dark/corrupted rendering due to inverted normals. Having the before/after diagrams was super helpful for debugging and coding this out.

Part 5: Edge split

We implemented splitEdge by treating the operation as a local rewrite of the two triangles incident to the selected interior edge. First, we return immediately for boundary edges. Then we collect the the 6 original halfedges around the two triangles, their 4 outside twins, 4 old vertices, 5 old edges, and 2 old faces. After that, we create the new elements needed for a split: 1 midpoint vertex, 3 new edges, 2 new faces, and 6 new halfedges.

The midpoint vertex position is set to the average of the two endpoints of the split edge: m = 0.5 * (v0 + v1). We then reassign all local halfedge connectivity (next, twin, vertex, edge, and face) so the old two triangles become four triangles around the new midpoint. This is all constant-time local work and does not depend on total mesh size.

The main issue for me was assigning halfedges for every vertex, edge, and face after rewiring. At first, We focused on setNeighbors(...) and assumed everything was fine, but we still got weird behavior when repeatedly splitting/flipping edges (occasional bad traversal and inconsistent selection in the viewer). The fix was to explicitly set all representative pointers at the end, even when some of them could technically stay the same. Once we did that consistently, the operation became stable under repeated edits.

Part 6: Loop subdivision for mesh upsampling

We implemented Loop subdivision in three phases. First, we computed all updated positions on the original mesh: old vertices store their Loop-updated values in Vertex::newPosition, and each old edge stores the future midpoint value in Edge::newPosition. Second, we implemented 4-1 refinement by splitting each original edge exactly once, then flipping only new edges that connect one old vertex and one new vertex. Finally, we copied newPosition into position for all vertices.

The most useful debugging trick was strict bookkeeping of connectivity metadata. In particular, subdivision was unstable until edge isNew flags and representative halfedge pointers were assigned consistently after split/flip operations. Repeated stress testing (running multiple Loop iterations in a row) was the easiest way to catch pointer or flag mistakes.

From our screenshots, repeated Loop subdivision makes the mesh progressively smoother while increasing triangle density. Sharp corners and creases are gradually rounded because Loop is an approximating smoothing scheme. This effect can be reduced by pre-splitting edges near important features so those regions get more local geometric support before smoothing.

On dae/cube.dae, repeated subdivision without preprocessing produced slight asymmetry. This happens because the initial cube triangulation has directional diagonal bias, so local neighborhoods are not perfectly balanced under Loop updates. To reduce this, we pre-processed the cube using symmetric edge edits (mirrored flips/splits across opposite faces) before subdivision. After preprocessing, the subdivided result is more balanced and visibly more symmetric, though a little bias may remain.
